### Таблица 2: Анализ вариантов публикации (Хостинг)

Ты абсолютно прав насчет GitHub Pages: это **статический** хостинг. Он не сможет безопасно хранить API-ключи от OpenRouter/Gemini (они будут видны в исходном коде страницы), а также может блокироваться политиками CORS при прямых запросах к сторонним API.

| Критерий | GitHub Pages | Vercel (Free Tier) | Netlify (Free Tier) | Cloudflare Pages (Free) |
| :--- | :--- | :--- | :--- | :--- |
| **Тип хостинга** | Статический (только HTML/CSS/JS). | Статический + Serverless Functions (бэкенд). | Статический + Serverless Functions (бэкенд). | Статический + Workers (бэкенд). |
| **Безопасность API ключей** | **Нет.** Ключи придется "зашивать" в JS, что небезопасно. | **Да.** Можно создать простую серверную функцию (API route) для скрытия ключа. | **Да.** Аналогично Vercel, есть Netlify Functions. | **Да.** Через Cloudflare Workers. |
| **Доступность из РФ** | Высокая (иногда бывают редкие замедления, но в целом работает). | Высокая (стабильно работает, домены `.vercel.app` обычно не блокируются). | Высокая (стабильно, домены `.netlify.app` доступны). | Очень высокая (одна из самых устойчивых сетей). |
| **Скорость развертывания** | Мгновенно при пуше в Git. | Мгновенно при пуше в Git (идеальная интеграция с GitHub). | Мгновенно при пуше в Git. | Мгновенно, но настройка Workers чуть сложнее для новичка. |
| **Стоимость** | 0 ₽ | 0 ₽ (хватит за глаза для нашего проекта). | 0 ₽ (хватит за глаза). | 0 ₽ (очень щедрые лимиты). |
| **Вердикт** | **Не подходит** для задачи с AI API (риск утечки ключа, CORS). | **Идеальный выбор.** Проще всего настроить, отличная документация, решает проблему CORS и ключей. | **Отличная альтернатива** Vercel, если вдруг с Vercel возникнут сложности. | Хорош, но порог входа чуть выше из-за специфики Workers. |

### Итоговая рекомендуемая архитектура (на основе анализа)

1. **Среда разработки:** Локальная (обычные текстовые файлы или VS Code). Я пишу код, ты создаешь файлы (`index.html`, `style.css`, `script.js`) и вставляешь код. Это самый надежный и бесплатный путь.
2. **AI Модель:** **OpenRouter** (бесплатные модели, например, `meta-llama/llama-3-8b-instruct` или `mistralai/mistral-7b-instruct`). Они отлично справляются с ролью "продающего агента" и не требуют привязки карты.
3. **Книжный API:** **Open Library API** (`https://openlibrary.org/`). Бесплатный, без ключей, отдает обложки, описания и названия.
4. **Хостинг:** **Vercel** или **Netlify**. Мы подключим твой GitHub-репозиторий к Vercel. Я напишу для тебя крошечный "Serverless Function" (бэкенд-прокси), который будет безопасно обращаться к OpenRouter, чтобы твой API-ключ не украли, а GitHub Pages здесь не справится.

### Структура страницы (подтверждение ТЗ)
*   **Header:** Узкий, текстовый логотип/название.
*   **Main (Flexbox/Grid):**
    *   *Left (33%):* Чат-интерфейс. История сообщений, поле ввода, кнопка отправки (стилизация под Telegram/CSV экспорт, как ты просил).
    *   *Right (67%):* Панель книг. Заголовок тематики, сетка карточек (12 шт. на страницу), вертикальный скролл внутри блока, пагинация.
*   **Footer:** Узкий, текстовый.
*   **Логика:** При клике на книгу открывается модальное окно или разворачивается блок с "произвольной рекламой" (генерируется AI на лету или берется шаблонный текст + данные из API).
